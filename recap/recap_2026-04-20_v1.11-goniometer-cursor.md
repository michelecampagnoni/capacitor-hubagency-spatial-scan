## Problemi affrontati

### 1. Goniometro non visibile in TOP mode
In TOP mode il goniometro veniva disegnato centrato sull'**ultimo punto confermato** a XZ
del pavimento, proiettato al soffitto. Ma il cursore arancione è su un **XZ diverso** (dove
la camera punta, tipicamente un angolo lontano). Il cerchio del goniometro (raggio 0.40m)
appariva lontano dal cursore → completamente fuori dal campo visivo quando si guarda in alto.

### 2. Parete preview non segue il cursore (fix difensivo)
`lastLivePreview` veniva impostato da `perimeterCapture.livePreview()` nella sezione reticle,
poi aggiornato solo al caso snap dal blocco goniometro. Aggiornamento esplicito post-goniometro
mancante nel caso no-snap, che potrebbe divergere se `SNAP_THRESHOLD` venisse modificato.

---

## Soluzioni

### Fix 1 — Goniometro centrato al cursore in TOP mode (`ScanningActivity.kt`)

Il goniometro viene passato al renderer con **XZ del cursore arancione** invece dell'XZ
dell'ultimo punto confermato:

```kotlin
goniometerCenter = if (!openingMode && ...) {
    val c = goniometerCenterPt
    if (topModeActive && c != null) {
        val tc = lastTopCursorWorld
        if (tc != null) floatArrayOf(tc[0], c[1], tc[2]) else c
    } else c
} else null
```

Effetto:
- **TOP mode**: cerchio goniometro centrato sul cursore arancione (XZ = proiezione soffitto)
- **FLOOR mode**: comportamento invariato (XZ = ultimo punto confermato)
- **Snap math**: invariato — il calcolo snap usa sempre `gonioCenterPt` (ultimo punto floor)

Comportamento visivo in TOP mode con fix:
```
Utente guarda in alto verso un angolo soffitto:
  - Cursore arancione = dove sta puntando
  - Goniometro ciano = cerchio intorno al cursore arancione (non lontano)
  - Settore ±65° attorno alla direzione orizzontale camera
  - Snap 1°: raggio ciano pieno che attraversa il cursore quando vicino a un asse

Prima del fix:
  - Goniometro centrato sull'ultimo punto (lontano dal cursore) → off-screen
```

### Fix 2 — Parete preview esplicita dall'output goniometro (`ScanningActivity.kt`)

Dopo il blocco goniometro + sync, aggiornamento esplicito di `lastLivePreview`:

```kotlin
val sr = lastSnappedReticle
if (gonioCenterPt != null && sr != null) lastLivePreview = sr
if (topModeActive && sr != null)
    lastTopCursorWorld = floatArrayOf(sr[0], lastFloorY + wallHeightPreview, sr[2])
```

`lastLivePreview` viene sempre aggiornato con l'output goniometro (snappato o raw), non
dipende più dalla sequenza di aggiornamento di `livePreview()` interno. Il risultato è
identico con `SNAP_THRESHOLD=0` (attuale), ma è corretto per qualsiasi valore futuro.
