## Flow UX proposto

```
[Esporta]
   ↓
[Dialog: Nome stanza]  →  conferma
   ↓
doStopScan(showContinueDialog=true)
   ↓
save history + deliver result to plugin
   ↓
[Dialog 1: "Salotto" salvata — altro ambiente?]
   ├── "No"               → finish()
   ├── "Sì, nuovo"        → startNextScan(seed senza apertura)
   └── "Sì, con collegamento"
          ↓
       [Dialog 2: lista porte/portefinestre del room appena salvato]
          ├── "Nessuna apertura" → startNextScan(seed senza apertura)
          ├── [seleziona porta] → startNextScan(seed con dims apertura)
          └── "Senza collegamento" → startNextScan(seed senza apertura)
                 ↓
             nuova ScanningActivity parte
             con pendingNextRoomSeed in companion object
```

### Separazione plugin vs UI path
- **UI path** (utente preme "Esporta"): `doStopScan(showContinueDialog=true)` → mostra dialog multi-stanza
- **Plugin path** (`requestStop()` da JS): `doStopScan()` → default `false` → `finish()` diretto, nessun dialog
  → Il JS caller non viene mai bloccato da un dialog non richiesto

### Sicurezza lifecycle `instance`
- `onDestroy` controlla `if (instance === this)` (reference equality)
- Quando la vecchia Activity avvia la nuova e poi chiama `finish()`:
  1. Nuova Activity → `onCreate` → `instance = newActivity`
  2. Vecchia Activity → `onDestroy` → `instance === oldActivity` → false → non annulla
- Nessuna race condition

## Come viene riusata l'apertura precedente

### `NextRoomSeed` (companion object di ScanningActivity)
```kotlin
data class NextRoomSeed(
    val fromRoomId:      String,   // id della stanza salvata
    val fromRoomName:    String,   // nome per connectionLabel ("← Salotto")
    val fromOpeningId:   String?,  // id apertura scelta, null = nessuna
    val suggestedKind:   OpeningKind?,
    val suggestedWidth:  Float?,
    val suggestedHeight: Float?,
    val suggestedBottom: Float?
)
```

### `spawnOpening()` — consumo del seed
```kotlin
val seed      = pendingNextRoomSeed?.takeIf { it.suggestedKind == kind }
val useWidth  = seed?.suggestedWidth  ?: kind.defaultWidth
val useHeight = seed?.suggestedHeight ?: kind.defaultHeight
val useBottom = seed?.suggestedBottom ?: kind.defaultBottom

val opening = OpeningModel(
    ...
    width                = useWidth,    ← dimensioni ereditate
    height               = useHeight,   ← dimensioni ereditate
    bottom               = useBottom,   ← posizione verticale ereditata
    isInternalConnection = seed != null,
    linkedRoomId         = seed?.fromRoomId,
    linkedOpeningId      = seed?.fromOpeningId,
    connectionLabel      = seed?.let { "← ${it.fromRoomName}" }
)
if (seed != null) pendingNextRoomSeed = null  // ← consumato, una sola volta
```

Regole:
- Il seed si applica **solo alla prima** apertura che corrisponde per `kind`
- Se il tipo non corrisponde (es. seed è DOOR, l'utente spawna WINDOW) → usa defaulti
- Dopo il consumo: le successive aperture usano i defaulti normali
- L'utente può sempre modificare le dimensioni manualmente nel pannello stepper

## Rischi di regressione
| Rischio | Probabilità | Note |
|---------|------------|------|
| `requestStop()` mostra dialog | Nessuno | Default `showContinueDialog=false`, nessuna modifica |
| Race condition `instance` | Nessuno | `onDestroy` usa `===` reference check |
| `spawnOpening` crash | Nessuno | `takeIf` restituisce null se kind non corrisponde |
| Dialog non dismettibile accidentalmente | Nessuno | `setCancelable(false)` su entrambi |
| Seed non consumato su cancel | Nessuno | `cancelScanAndFinish()` non chiama `doStopScan()` → il seed rimane ma è sovrapposto dalla prossima chiamata |
| Motore capture | Zero | Nessun file del capture motor toccato |
